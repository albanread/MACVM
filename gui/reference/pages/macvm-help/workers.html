Worker VMs & OTP Supervision

MACVM's answer to concurrency is not green threads inside one heap but multiple whole VMs — each with its own heap, JIT and GC, on its own OS thread — talking to each other only by copying messages. On top of that substrate sits an OTP-style supervision layer: named services, restart policies, and a deadline-bounded request/reply protocol, so a worker crashing is an ordinary, recoverable event rather than something calling code has to guard against by hand.

Workers: share-nothing parallelism

The primary VM (the one this help page is running in) can spawn worker VMs and exchange messages with them by deep copy — never by reference. No shared heap, no shared globals, no identity across a VM boundary: Erlang-style share-nothing parallelism, driven entirely from Smalltalk.

(Worker spawn: 'Worker onMessage: [:m |
    Worker reply: m payload]')
    send: 42
    onReply: [:r |
        Transcript showCr: 'got ', r printString].

Every send is fire-and-forget; a reply (if any) runs later as a continuation (send:onReply:), never by blocking — the primary is never waiting on any one worker, only asleep until the next envelope wakes it. The Demos menu's “Mandelbrot — parallel workers” computes every frame in bands across 4 worker VMs this same way. Full design: docs/multi-smalltalk-worker.md.

Supervision: let it crash, then recover

MACVM has no exception system — self error: stops the current computation outright. Rather than build scoped catch handling (deliberately rejected — see docs/design_philosophy_roadmap.md), the worker model answers the same problem the way Erlang/OTP does: a crashed worker is reported as an ordinary message, and a supervisor restarts it by policy. Full design: docs/otp_workers_design.md.

  • WorkerSupervisor — owns a set of named children, starts them in order, and reacts to a child's crash (never a deliberate terminate, which reports no death at all) by its strategy: #oneForOne restarts just the dead child, #oneForAll restarts every sibling, #restForOne restarts the dead child and everything started after it. A supervisor can itself be a child of another supervisor, building a tree; a child that exhausts its own restart budget (intensity: N restarts within a time window) escalates — its parent treats the give-up exactly like an ordinary child death.
  • WorkerNames — a name registry (WorkerNames named: #sieve), rebound on every restart, so calling code never holds a handle to a corpse.
  • ServiceWorker — the request/reply face over a named worker: cast: for fire-and-forget, call:timeoutMs:onReply:onError: for a deadline-bounded request whose every failure mode — a timeout, the target's death, or an RPC-shaped error — lands in one onError: funnel. Nothing ever blocks: a deadline is just an entry a periodic sweep retires, on the same cadence the worker inbox already wakes on.
(WorkerSupervisor
    named: #services strategy: #oneForOne)
    superviseNamed: #echo
        init: 'Worker onMessage: [:m |
            Worker reply: m payload]';
    start.

(ServiceWorker named: #echo)
    call: 42 timeoutMs: 2000
    onReply: [:r |
        Transcript showCr: 'echoed: ', r printString]
    onError: [:e |
        Transcript showCr: 'failed: ', e printString].

The IoWorker (world/62_ioworker.mst) is the first real service built this way: a dedicated worker multiplexing many file descriptors through one kqueue, so no other VM ever blocks on I/O. Its kernel-level watch registrations live in the primary, not the worker — so when a supervised IoWorker crashes and restarts, no watch needs to be re-registered; an afterRestart: hook just rebinds the fresh worker handle and resumes polling.
