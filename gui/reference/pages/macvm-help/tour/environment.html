The Programming Environment

The window you are reading this in is the whole environment. It is a native Cocoa NSWindow holding a single WKWebView, and nothing else. Only the top menu bar is native AppKit; the toolbar across the top and the status bar along the bottom are plain HTML, injected into every page as it loads. So a "screen" in MACVM is just an HTML document — like this one — and the environment is the set of documents you can navigate between, edit, and run.

This is Strongtalk's live-page idea, kept almost verbatim. A document is static HTML until you reach for one of two extension tags, and then it becomes live.

The two live tags

The first is <a doit="...">. It looks like an ordinary link, but instead of navigating it hands its Smalltalk source to the embedded VM and evaluates it — for real, on the same object world you are running. This one is fully wired. Here it is doing arithmetic and printing to the Transcript:

  • Show 6 * 7 — evaluates 6 * 7 and writes the answer to the Transcript.

  • Show a labelled sum — string concatenation with , around a real integer result.

There is no compile step you have to think about: the optimizing compiler is underneath every one of these, so a hot doit is native code just like everything else. A doit is the smallest unit of live programming in the system — a link that is also a workspace.

The second tag is <smappl visual="...">, which is meant to embed a live widget inline in the page — a button, a field, a small view built from Smalltalk. Be warned: today this is only a placeholder. The class browser

The browser is wired, and it is where real work happens. You pick a package, then a class, then a method, and its source appears in an editable pane. Change the source, accept it, and the method is live-compiled into the running world — the next send goes to your new code, with the adaptive compiler recompiling it as it heats up, and any stale optimized code deoptimized out from under itself so nothing runs a version that no longer exists.

Crucially, that edit does not vanish when you quit. It is persisted to the image store, so the world you build stays built.

The image store, and booting from it

The image store is a SQLite database that holds the object world as editable rows — classes, methods, source. You can edit it offline, outside a running VM, which is a genuinely useful thing to be able to do to a Smalltalk image. On startup a DB→VM boot loader reconstructs the live world from that database, and it does so byte-identically to booting from a classic .mst source image — verified by a differential test, not by hope. The database is the source of truth, and the browser is one live editor onto it.

Themes

The native Theme menu restyles the whole environment without touching a single page. Classic is the period-accurate Strongtalk look — pixel icons, Win95-style bevels. Hi-Def is a flat, modern skin. There are others besides — dark, Squeak, CRT green and amber, an Alto-style monochrome. Every theme is just a stylesheet swapped into the same WKWebView, and every one renders these exact same pages. The content is HTML; the look is a menu choice.
