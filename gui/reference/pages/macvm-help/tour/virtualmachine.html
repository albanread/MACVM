The Virtual Machine

Before the clever part — the adaptive compiler on the next page — there has to be an object world for it to run on, and a garbage collector underneath that. MACVM keeps these deliberately plain, because everything interesting later depends on them being correct. Two choices here are worth dwelling on: how an object is represented, and how the heap is collected while compiled code is running on top of it.

The object model

MACVM is class-based, in the Strongtalk line. Every object is an instance of a class, and behaviour lives in the class, not in the object. Self's prototype-and-map model — where an object carries its own map and clones its prototype — was evaluated and deliberately set aside. The class model is simpler to reason about, and it is what Smalltalk programmers already expect.

An object reference is a direct tagged pointer. The reference is the object's machine address, with information packed into its low bits — a 2-bit tag that tells a SmallInteger (an smi, whose value rides inside the pointer itself) apart from an ordinary object pointer (an oop). So small integers are immediates: they never touch the heap, never allocate, and never trouble the collector. The arithmetic below runs entirely in tagged pointers — no object is created to hold the result: Run 6 × 7

A heap object begins with a two-word header, [mark][klass]. The mark word carries the garbage collector's and identity bookkeeping; the klass word points at the object's class. That is the whole overhead — two words, then the instance's own fields.

Why no object table

This is the choice that pays for itself later. Smalltalk-80 and many classic systems place an object table between a reference and the object: every reference is an index into a table, and the table entry holds the real address. That indirection is genuinely convenient — it makes become: nearly free and lets the collector relocate an object by rewriting a single table slot. But it charges a memory indirection on every field access, forever.

MACVM has no object table. A pointer is the address, so reading a field is a single load. The bill for that speed comes due in exactly one place: when the collector moves an object, there is no table slot to patch — it must find and rewrite every pointer to that object, wherever it lives. That is the hard part, and it is where the next section goes.

The garbage collector

There are two collectors, of a familiar shape. A generational scavenge handles the young generation, on the usual bet that most objects die young; a full compacting collector sweeps and compacts the whole heap when it must. Nothing exotic in the algorithms themselves. The interesting engineering is when they are allowed to run.

Both of them run underneath live, moving compiled frames. This is the sentence that matters. When the optimizing compiler has native code on the stack, that code is holding oops in registers and stack slots for speed. If the collector then relocates the objects those references point at — and a compacting collector relocates almost everything — every one of those live references, down in the CPU registers of a running method, has to be found and updated. Precisely. Miss one, or mistake a raw integer for a pointer, and the system dies.

Collecting underneath live, moving code

Two mechanisms make that safe:

  • Precise oop-maps. At every safepoint where a collection can happen, the compiler has recorded exactly which registers and stack slots hold oops and which hold raw non-pointer bits. The collector reads that map, so it updates real references and never touches a raw integer.

  • A mixed-tier frame walker. A real stack is not uniform — it interleaves interpreter frames with compiled frames from possibly different methods and tiers. The walker understands the whole mixture and visits the roots in each frame correctly, whichever tier produced it.

The payoff is precisely the freedom the optimizer needs: it is safe for compiled code to keep values in registers across a GC safepoint. Without precise maps you would have to spill every live value to a fixed format before any collection, or forbid the collector from running in the middle of a method at all — and either one throws away the register residency that makes native code fast. With them, hot values stay in registers and the collector is free to move the heap out from under a running method. The plain object model up top is what lets these two pieces stay honest about what is and isn't a pointer.

That is the foundation. The next page is what stands on it: the adaptive compiler that fills those registers, and how it gives them up again — safely — when a speculation turns out to be wrong.
