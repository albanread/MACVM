How MACVM Relates to Strongtalk

MACVM is a re-implementation from Strongtalk's published design, not a fork of its source. So it is worth being precise about the family resemblance — what carries over unchanged, what was deliberately left behind, and what is genuinely new. One thing runs through all three lists without qualification: the language. The syntax and the basic semantics are Smalltalk-80 throughout, exactly as they were in Strongtalk.

What MACVM keeps

  • The class model. Direct tagged pointers, a two-word [mark][klass] header, a two-bit tag for smi vs oop, and no object table — Strongtalk's object model, kept intact. Self's prototype/map model was evaluated and set aside.

  • Adaptive optimization. Inline caches, PICs, type feedback, method and block inlining, per-klass customization, and correct deoptimization when a speculation breaks — the Self → Strongtalk engine, reproduced and extended.

  • The live-HTML environment. Pages that execute Smalltalk (doit) and embed widgets (smappl). That is the Strongtalk "live page" idea, and it is the surface you are reading this tour on.

  • The fast-float ambition — now delivered. Strongtalk sketched an experimental fast-float scheme and left it unfinished. MACVM finished it: a mono-Double send site compiles to a guarded unbox, native fmul/fadd/fcmp, and a box only where a boxed value is actually observed. See the Fast Floating Point page.

What it drops, or has not built yet

  • The optional static type system. Strongtalk's most famous idea — a pluggable, provably-sound type layer that never changed run-time semantics — is not implemented in MACVM. It may come one day; it is not here.

  • Mixins. Strongtalk's mixin-based inheritance is not implemented. MACVM uses ordinary single inheritance.

  • The glyph-based flyweight UI. Strongtalk drew its whole interface from lightweight glyphs. MACVM does not — it hosts a native Cocoa window around a single WKWebView and renders the interface as HTML.

  • Native / green threads with async C callouts. Not yet. The FFI is synchronous Tier-1 POSIX (dlsym + shape-keyed trampolines + an Alien raw-memory type); Tier-2 Cocoa dispatch and concurrency are future work.

What is new

  • Rust. The whole VM is new Rust — interpreter, optimizing compiler, garbage collector — owing nothing to Strongtalk's source code.

  • Apple-Silicon-first. arm64 throughout, with a hardened JIT (MAP_JIT / W^X) and pointer-authentication awareness. Not a port of an x86 design.

  • A SQLite image store. Offline image editing plus a DB → VM boot loader that reconstructs the world byte-identically to a classic image boot.

  • Measured reliability. The interpreter is a differential oracle: compiled output must be byte-identical to interpreted output, gated against GC and deopt stress matrices. Dependability is treated here as a first-class goal, right alongside speed.

  • Near-total compiled coverage that stays compiled. About 98.7% of the methods that run compile; on real workloads 98.6—99.8% of executed work runs as compiled native code, closures included — and stays that way.

The one thing that has not moved an inch is the language itself. A live link, to prove it: this runs ordinary Smalltalk-80, evaluated left-to-right through binary operators just as you would expect — Run 3 + 4 * 2 — which prints 14, not 11.
