SIMD Vector Arithmetic

The previous page took a single Double out of its box and did the arithmetic in a register. SIMD is that same idea one register width up. The Apple Silicon NEON unit works on 128-bit vectors — two 64-bit lanes, or four 32-bit lanes — and a single instruction operates on every lane at once. MACVM exposes that directly: a value that is a bundle of lanes, whole-vector arithmetic on it, and the same operations streamed over arrays.

Vector values

A small family of immutable value classes, one per 128-bit NEON arrangement:

  • Float64x2 — two 64-bit floats (.2d);

  • Float32x4 — four 32-bit floats (.4s);

  • Int32x4 — four 32-bit integers (.4s); arithmetic wraps on 32-bit overflow, and there is no vector integer divide.

Each is a fixed 16-byte value object whose body the garbage collector never scans — exactly the treatment a boxed Double already gets, just eight bytes wider. It is boxed at rest (128 bits will not fit in a 64-bit tagged word) and register-resident only inside a compiled region. The protocol reads the way you would hope:

  • Float32x4 splat: 1.5 — broadcast one value to all four lanes; Float32x4 x: 1.0 y: 2.0 z: 3.0 w: 4.0 — a lane constructor.

  • v1 + v2, v1 * v2, v1 min: v2, v1 sqrt — elementwise, one NEON instruction each.

  • v1 at: 2 — extract a lane; v1 sum, v1 dot: v2 — horizontal reductions.

When the inline cache at an arithmetic send has only ever seen one vector class, the compiler fuses the whole send — guard, unbox with a single ldr q, the native fadd/fmul over the arrangement, and the re-box — into a short straight run of NEON. It measures roughly 13–15× over the interpreter.

Honest about the lanes

Elementwise operations keep the same parity discipline as the scalar float path. Lane i of Float64x2 + Float64x2 is exactly Double + Double for lane i, so a four-wide Float32x4 + is bit-identical to four scalar Float32 +s. The goldens stay sharp, and (as with the scalar fast-multiply-add decision) elementwise ops deliberately use the non-fused instructions so the per-lane result never rounds differently.

Reductions are the one place where honesty is required. Combining lanes with a pairwise tree reorders the additions, and floating-point addition is not associative — so a floating sum is not bit-identical to a sequential left-fold. MACVM does not pretend otherwise: a reduction is its own operation with a defined lane-combination order (verified against that definition, not against a scalar loop), and a separate sequentialSum is available when the exact scalar result is what you need. Integer reductions have no such caveat — integer addition is associative, so Int32x4 sum is exact regardless of order.

Streaming over arrays

Above the individual vectors sits FloatArray: a contiguous, GC-skip buffer of floats, and a set of bulk kernels that run whole-array operations as NEON loops — a +@ b (elementwise sum), a scale: k, a dot: b, a sum, a max, a min. These are written as explicit NEON intrinsics, on purpose: a bulk kernel should drive the silicon deliberately, not be a scalar loop left to an optimizer to maybe vectorize — MACVM has its own compiler and does not hope. Because the whole loop is a single primitive, the interpreter and the JIT run the identical code.

Where it pays

The vector path drops straight in wherever the work is lane-independent: array add, multiply and scale, dot products, colour-matrix and coordinate transforms, DSP filters, and packing a pixel buffer four pixels at a time. It does not drop in where the control flow is data-dependent — the Mandelbrot escape loop, for instance, has a per-pixel trip count, so doing four pixels at once needs a lane mask that retires escaped lanes one at a time. That is a genuine algorithm rewrite, not a free win, and it is left as an advanced path rather than pretended into the value model here.
