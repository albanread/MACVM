Conclusion

MACVM is a working research virtual machine. The core is done: the object world, the dispatch-based bytecode interpreter, generational scavenge plus a full compacting collector running underneath live, moving compiled frames, and the adaptive optimizing compiler — inline caches, customization, method and block inlining, deoptimization, on-stack replacement. Around that core sit a Tier-1 POSIX FFI and the full Cocoa bridge (Objective-C objects as ordinary Smalltalk receivers, AppKit across the main-thread hop, button callbacks as blocks — see Cocoa from Smalltalk), multi-VM concurrency (a primary plus worker VMs on their own threads, deep-copied messages), GUI embedding through a crash-surviving VmHandle, a SQLite image store with a byte-identical DB → VM boot loader, and full closure compilation — literal blocks that splice inline rather than allocate.

Two theses are settled. Compiler coverage is achieved — almost everything that runs, runs compiled, and stays compiled. And the reliability thesis is realized: the VM is fast and stable at the same time, which is exactly the combination the released Strongtalk could never hold. deltablue runs about 53x faster under the compiler than under the interpreter, and a scavenge or a full compaction can run underneath that compiled code without disturbing it. Fast was never the hard part. Fast-and-stable is, and that is the part that works.

Still open

  • An in-image inspector — a live object inspector written in Smalltalk. (Class mirrors and browsers already live in the image — ClassMirror, ClassHierarchyOutliner, ClassOutliner — beside their Rust counterparts, and smappl widgets now render live rather than as placeholders.)

  • An optional static type system — Strongtalk's, if it is ever built here.

Not on that list, by choice: green threads. MACVM's concurrency model is multiple real VMs, each on its own OS thread — a fresh VM thread starts in well under 25 ms on modern hardware, so a second (or tenth) VM is cheap. Cooperative threads inside a single VM would mean a scheduler, hand-rolled stacks, and a whole second story about async C callouts, all to approximate what real OS threads already give us. So it is a non-goal, not a to-do.

Strongtalk showed how much was possible at once: a Smalltalk that ran at serious speed, with an adaptive optimizer, an optional type system, and a live programming environment, all in one system — a remarkable body of work. MACVM is a wager that, with modern tooling and relentless differential testing, those same ideas make a foundation you can build on and keep building on. It is still research, and the list above is real. But the thing runs, it runs fast, and it keeps running — the adaptive-optimization dream, carried forward: Sign off.
