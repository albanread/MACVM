Fast Floating Point

The introduction promised a page that finishes something. This is it. Strongtalk shipped an experimental fast-float scheme and was honest that it was unfinished: it leaned on a special <FloatVal> annotation that was never integrated with the type system, it made the programmer write the box and unbox by hand, and it could not carry an unboxed double across a method boundary. A sketch of the right idea, left on the bench.

MACVM finishes it, and does so without asking the programmer for anything. There is no annotation. You write ordinary Smalltalk float arithmetic; the compiler does the rest.

Float regions

The mechanism is the float region. At a send site where the inline cache has only ever seen Double, the IC is the type oracle — that is why no <FloatVal> annotation is needed. The compiler emits a guarded unbox, then native fmul / fadd / fcmp, and a re-box only where a boxed value is actually observed. Inside the region there is no allocation, no GC interaction, and no message send at all — just floating-point instructions.

A few pieces make that hold together:

  • a second register file — unboxed floats live in d0–d7 scratch plus a d8–d15 write-through residency tier, independent of the integer register allocator. A raw f64 is invisible to the moving GC: it is never in an oop map, so the collector can run underneath the loop and simply never sees it;

  • a box/unbox reducer — FUnbox(FBox x) cancels to x, dead boxes are eliminated, and boxing that is only needed on the deopt path is sunk there. Float-temp promotion then lifts a temporary that always holds a Double to a raw f64 that lives across the whole loop;

  • one new deopt-map kind, DoubleSlot, which tells the materializer “this stack slot is raw float bits — box it back” if we ever have to bail to the interpreter. That is what lets an unboxed double survive across a method boundary that Strongtalk could not cross;

  • libm transcendentals — sin, cos, tan, exp, ln, atan, sqrt wired in as primitives, so a real numeric kernel never has to leave the region.

Measured, not proposed

The proof is the GUI's Mandelbrot demo — 420 × 220, release build, on a fanless Apple-Silicon MacBook Air. Each layer above removes a whole category of cost, and you can watch it fall:

StageTimeAllocation / render
boxed sends (baseline)746 ms708 MB
pixel-buffer output458 ms595 MB
float-region fuse180 ms595 MB
sunk boxing + temp promotion166 ms4 MB
strength-reduced coordinates38 ms0
d-register residency25 ms0

That is roughly 30× end to end, and the last two rows are the interesting ones: zero allocation, zero deopts, one scavenge-free heap per render. The inner loop touches neither the allocator nor the collector. There is also a hand-ported C version of the identical kernel, built with the same clang and taken as the best of thirty runs, kept as an external yardstick — and the final MACVM figure lands right up against it. The point is not the exact numbers; it is that ordinary Smalltalk float code, with no annotations, reaches the neighbourhood of hand-written C.
