The Optimizing Compiler

Here is the heart of the thing. MACVM runs on two tiers. The first is a simple dispatch-based bytecode interpreter — a fetch–decode–match loop that starts instantly and asks no questions. The second is a tier-1 optimizing JIT that compiles hot methods to native arm64 code. The strategy is the one Self pioneered and Strongtalk carried into a class-based world: run first, then recompile. You never pay to optimize code that only runs once, and by the time a method is hot you know something the interpreter had to discover the hard way — what types actually flow through it.

Type feedback: the inline cache is the oracle

Every send site starts monomorphic. The first time anObject foo executes, the interpreter records the receiver's klass and the method it dispatched to, right there at the call site — an inline cache. Hit the same klass again and the lookup is free. When a second klass shows up, the inline cache grows into a polymorphic inline cache (a PIC): a small guarded list of the klasses actually seen here, in this program, on this data.

That PIC is not just a dispatch accelerator. It is a record of measured fact. When the JIT compiles the method, the PIC tells it exactly which receiver types occur at every send — so the compiler never has to guess, and never has to fall back to a slow general lookup for the cases that never happen. The inline cache is the type oracle. No declarations, no annotations; the running program is the specification.

Inlining, customization, devirtualization

Armed with that feedback, the compiler does the aggressive things:

  • Method inlining — a monomorphic send whose target is known is replaced by the target's body, turning a call into straight-line code and exposing everything inside it to further optimization.

  • Block inlining — literal blocks do not become heap-allocated closures that get invoked later. They splice inline at the point of use, including multi-basic-block blocks that carry a non-local ^ return out of an enclosing method. A whileTrue: or an ifTrue:ifFalse: compiles to exactly the branch you would have written by hand.

  • Per-klass customization — a method is compiled once per receiver klass. Inside the customized copy the receiver's type is a constant, so its own self sends resolve statically and inline in turn.

  • Devirtualization — and so the self-sends and the block arguments that Smalltalk leans on for everything stop being dynamic dispatch. A customized method's self foo becomes a direct call or an inline body; a block passed as an argument is invoked at a known site rather than through a general value-send.

Why speculating is safe: deoptimization and OSR

All of this is speculation. The compiler bet that the klasses it saw are the klasses it will keep seeing. Sometimes that bet is wrong — you redefine a method, or a receiver of a klass this site never witnessed finally turns up. The reason MACVM can speculate so aggressively is that it can always bail out correctly.

When a guard fails, the running native frame is deoptimized: the compiled activation is reconstructed as interpreter frames, live values back in their proper slots, and execution continues in the interpreter exactly as if the method had never been compiled. No wrong answer is ever produced — the fall-back is not a fast approximation, it is the same computation. The reverse door is on-stack replacement (OSR): a loop that turns out to be hot while it is already running can be entered in compiled form mid-flight, without waiting for the next call. Deopt is how you leave the fast tier safely; OSR is how you enter it early.

Because bailing back is always correct, the compiler is free to be greedy. Speculation costs nothing when it is right and nothing but a recompile when it is wrong.

Compiler coverage achieved

This is the milestone. The goal was never "optimize the inner loop of a benchmark" — it was to make the optimizing compiler carry essentially all of a real program. It does.

  • ~98.7% of the methods that actually run get compiled to native code. The remainder are native primitives, which are already native — they lose nothing by staying that way.

  • On real workloads, 98.6–99.8% of executed bytecode-work runs as compiled native code — closures included — and stays compiled. The system reaches steady state and holds it, rather than thrashing between tiers.

And the speedups, interpreter versus JIT, measured on a fanless MacBook Air:

  • deltablue — 214 ms → 4 ms, about 53×.

  • richards — ~205 ms → ~6 ms, about 34×.

  • sieve — 88 ms → 9 ms, about 10×.

  • ctxloop (a closure/OSR microbenchmark) — 134 ms → 1 ms, about 134×.

The closure microbenchmark is the interesting one: it is almost entirely block invocation and a hot loop, exactly the shape that a naive compiler leaves on the table. Splicing the blocks inline and entering via OSR is what turns 134 ms into 1.
