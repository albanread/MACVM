A Tour of MACVM

Introduction

MACVM is a virtual machine for Smalltalk, written from scratch in Rust for macOS on Apple Silicon (arm64). It is in the Self → Strongtalk lineage: a class-based object model with an adaptive optimizing compiler driven by type feedback. It runs a real object world, and it runs it fast — on the standard benchmarks the optimizing compiler owns essentially all of the runtime.

This page you are reading is itself a MACVM document: a plain HTML file, shown in a native window, extended with two live tags — a link that executes Smalltalk instead of navigating, and a tag that embeds a live widget inline. That is the Strongtalk "live page" idea, and it is where this tour comes from.

The system that inspired it

Strongtalk was a fundamental redesign of Smalltalk-80, developed in secret at Animorphic Systems in the mid-1990s and later released by Sun as unsupported software. Its virtual machine executed Smalltalk far faster than any implementation before it, using adaptive optimization — run first, then recompile hot code with type feedback — the technique the Self group pioneered and that later became the core of Sun's production Java VM.

The released Strongtalk was fascinating, complex, and completely unstable. It ran well enough for a few minutes before crashing: a wonderful ten-minute demo, and unusable for anything more. It is easy to spend a great many enjoyable hours in a debugger trying to stabilize code like that and never succeed — the ideas were sound, but the artifact could not be made to stand up.

MACVM is the other approach. Decades later, with modern tooling, it is far easier to re-implement a design from its documentation than to repair the original. So MACVM is built to a large extent on Strongtalk's published design and its excellent documentation, cheating from that record wherever cheating is possible — while owing nothing to its source code. The bytecode interpreter and the optimizing compiler are new Rust; the assembler is reused from a sister project; the garbage collector, unavoidably, is new.

What this tour covers

Like the Strongtalk tour it is modelled on, this document assumes you know Smalltalk and dives into the interesting parts:

  • a two-tier engine — a simple dispatch-based bytecode interpreter that starts instantly, and a tier-1 optimizing JIT that recompiles hot code with type feedback and deoptimizes safely when its assumptions break;

  • near-total compiled coverage — almost every method that runs is compiled to native code, closures included, and stays compiled;

  • fast floating point that fulfils the experimental promise Strongtalk sketched but never finished — measured, not merely proposed;

  • a garbage collector that runs underneath live, moving compiled frames, which is what makes all of the above safe;

  • and an almost absurd level of introspection and debugging, backed by extensive tests — because the one lesson Strongtalk teaches most clearly is that fast-but-crashes is a demo, not a system. Reliability is the whole point.
