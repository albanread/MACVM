Cocoa from Smalltalk

MACVM can talk to macOS directly: Foundation and AppKit objects are ordinary Smalltalk receivers. You look a class up once, and from there Objective-C messages are just keyword sends — the selector text is identical in both languages.

    s := (Cocoa classNamed: 'NSMutableString') alloc init.
    s appendString: 'hello'.
    s length.                "→ 5"
    s asString.              "→ its -description, as a Smalltalk String"
    s release.

How it works (the two-sentence version)

A Cocoa object lives in Smalltalk as an ObjcRef — a tiny wrapper holding one retained reference; the two memory managers never see each other's pointers. Any message an ObjcRef doesn't itself understand is forwarded to Objective-C, with the argument and return types read from the live runtime's own method signatures — so integers, floats, booleans, strings, and the small structs (NSRange, CGPoint, CGRect) all convert automatically, in both directions.

What converts to what

  • Smalltalk String arguments become temporary NSStrings; NSString-returning selectors used via sendString: / asString come back as Smalltalk Strings.
  • Integers, Doubles, and true/false map to the declared C types — passing 3 where a double is declared does the right thing.
  • NSRange results are 2-element Arrays {location. length}; CGPoint/CGRect are Arrays of 2/4 Doubles. Struct arguments are written the same way: a frame rectangle is (Array with: x with: y with: w with: h).
  • Everything else comes back as another ObjcRef.

Ownership: release, poolDo:, and the safety bias

Every wrapper owns exactly one reference. Send release when you are done; a released wrapper is poisoned — later sends raise cleanly, and a double release refuses. Cocoa's naming conventions are honored automatically: alloc, new, copy, and init results arrive already-owned, and init invalidates the receiver it consumed (always use init's result). For bursts of temporaries, let the pool do the releasing:

    s := Cocoa poolDo: [:pool |
        (Cocoa nsString: 'scratch').          "swept on exit"
        pool keep: (Cocoa nsString: 'mine')]. "survives; release it yourself"

The bridge always errs toward a leak, never a double-free: a forgotten release wastes a few bytes; manual reference counting (retain/dealloc…) is refused outright. An NSException is caught and re-raised as an ordinary Smalltalk error — the VM never dies.

AppKit: onMain and actions

AppKit is main-thread-only, and the VM runs on its own thread. Prefix any UI send with onMain and it runs on the main thread, synchronously:

    win onMain makeKeyAndOrderFront: nil.

Buttons call back the other way: Cocoa action: turns a block into a target/action pair. The click is queued and your block runs on the VM thread between doits — callbacks never interrupt running Smalltalk.

    act := Cocoa action: [ Transcript showCr: 'clicked!' ].
    btn onMain setTarget: act.
    btn onMain setAction: 'macvmFire:'.

CocoaPad — the demo

Demos → CocoaPad builds a native window — an NSWindow, a text field, and a button — entirely in Smalltalk (world/50_cocoapad.mst, class CocoaPad; browse it!). Click Say hello and the round trip completes: AppKit fires the action on the main thread, the event crosses to the VM thread, CocoaPad clicked runs, and it answers by writing into the text field back through the main-thread hop. Everything above, in about sixty lines.

Name collisions and escapes

A few selectors are Smalltalk's own and never reach Objective-C: class (use objcClass), hash, printString, isValid, release, and =/== (wrapper identity — use isEqual: for Cocoa equality). For those, or any unusual shape, the explicit form is always available: ref send: 'class' or ref send: 'sel:' args: {…} ret: #i64.

The full internal design — memory model, thread rules, exception handling — is in docs/cocoa_bridge_design.md.
