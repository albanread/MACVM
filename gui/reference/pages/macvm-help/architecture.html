MACVM Architecture

MACVM is a research virtual machine for macOS on Apple Silicon, in the Self → Strongtalk lineage: a class-based object model with an adaptive optimizing compiler driven by type feedback.

It is not a port of either VM. It takes the adaptive-optimization machinery both share — inline caches, polymorphic inline caches (PICs), type feedback, and deoptimization — and combines that with Strongtalk's representation: classes with direct tagged pointers and no object table, rather than Self's prototype/map model.

Goals

  • Class-based object model — Strongtalk-style classes, direct tagged pointers, a two-word [mark][klass] header, and no object table.
  • Adaptive optimization — start executing quickly with a baseline interpreter, then recompile hot code using type feedback gathered from polymorphic inline caches, the technique Self pioneered and Strongtalk and HotSpot both inherited.
  • Apple Silicon first — arm64 calling convention, W^X / JIT hardening (MAP_JIT, pthread_jit_write_protect_np), and pointer-authentication awareness, rather than a portable-first design retrofitted for arm64.
  • Native macOS integration — this GUI shell is part of that: a direct objc_msgSend bridge to Cocoa/AppKit/WebKit, the same approach already proven in sibling projects, rather than a cross-platform toolkit.
  • Multi-VM parallelism — Smalltalk code spawns worker VMs (each with its own heap, JIT, and GC on its own OS thread) and exchanges messages with them by deep copy — Erlang-style share-nothing parallelism, no shared state, no identity across heaps. Replies run as continuations (send:onReply:); the primary never waits and never polls. The Demos menu's “Mandelbrot — parallel workers” computes every frame in bands across 4 worker VMs. An OTP-style supervision layer sits on top — named services, restart policies, crash recovery — MACVM's answer to having no exception system: let it crash, then a supervisor restarts it. See Worker VMs & OTP supervision, docs/multi-smalltalk-worker.md and docs/otp_workers_design.md.

Where things live

The source is organised as module directories, not flat files:

  • src/oops/ — the object model: tagged pointers, headers, classes.
  • src/memory/ — object memory, allocation, and the garbage collector: generational scavenge plus the full compacting collector.
  • src/interpreter/ — the simple dispatch-based bytecode interpreter: a fetch-decode-match loop (the baseline tier).
  • src/bytecode/ — the bytecode set, its decoder, and the control-flow graph.
  • src/compiler/ — the optimizing compiler and its JASM AArch64 backend.
  • src/codecache/ — the native code cache, the stubs, and the deoptimization trap.
  • src/runtime/ — dispatch, activation frames, the deopt materializer, OSR, recompilation, and the debugger.
  • src/frontend/ — the .mst parser.
  • src/embed.rs — VmHandle, the embedding API the GUI drives.
  • world/ — the object world / image sources.
  • gui/ — this GUI shell: plan, reference pages, CSS — the very tree these pages live in.
  • docs/ — the full design record: SPEC.md, SPRINTS.md, DESIGN.md, and more.

The two-tier split is invisible from the outside: the interpreter and the JIT must agree on every answer, so which one ran is never observable. Click to evaluate a little arithmetic in the VM behind this viewer — 6 * 7
